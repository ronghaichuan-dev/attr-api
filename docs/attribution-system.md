## 1. 归因概述

### 什么是归因

移动归因是指识别哪个营销渠道、广告活动或触点驱动了用户的安装和后续行为（注册、购买等），将用户行为与具体的广告来源建立关联。

### 为什么需要归因

- **ROI 衡量**：精确衡量每个渠道、每条广告的投入产出比
- **预算优化**：识别高效和低效渠道，合理分配预算
- **效果优化**：为广告平台优化算法提供数据基础
- **反作弊**：识别虚假安装和流量欺诈

### 主流归因平台（MMP）

| 平台 | 特点 |
|------|------|
| **AppsFlyer** | 市场份额最大，Protect360 反作弊能力强 |
| **Adjust** | 隐私优先，SDK 签名防欺诈 |
| **Branch** | 深度链接能力突出，覆盖 6000+ 边缘场景 |
| **Singular** | ROI 分析和广告花费归因整合 |
| **Kochava** | 灵活的自定义归因能力 |

## 2. 归因模型

### 2.1 单触点模型

| 模型 | 归因逻辑 | 适用场景 |
|------|---------|---------|
| **末次点击 (Last-Click)** | 100% 归因到转化前最后一次点击 | 行业标准，移动端默认模型 |
| **首次点击 (First-Click)** | 100% 归因到第一次交互 | 衡量品牌曝光和漏斗顶端效果 |

> 末次点击是移动归因的主流模型，几乎所有 MMP 默认使用此模型。

### 2.2 多触点模型 (MTA)

| 模型 | 归因分配 |
|------|---------|
| **线性归因** | 所有触点平均分配 |
| **时间衰减** | 越接近转化的触点权重越高 |
| **U 型归因** | 首次 40% + 末次 40% + 中间 20% |
| **W 型归因** | 首次 30% + 线索创建 30% + 末次 30% + 其余 10% |

> **局限性**：多触点模型依赖用户级别的跨触点可见性，iOS ATT 隐私政策后可用性大幅降低。

### 2.3 展示归因 (View-Through Attribution, VTA)

用户**看到**广告但未点击，之后完成转化时的归因。特点：
- 归因窗口比点击短得多（通常 1-24 小时）
- 点击归因优先级始终高于展示归因

## 3. 归因窗口

归因窗口（Lookback Window）是指广告交互后，后续转化可被归因到该交互的时间范围。

### 3.1 行业标准默认值

| 窗口类型 | 默认时长 | 说明 |
|---------|---------|------|
| **点击归因窗口** | 7 天 | 行业标准 |
| **展示归因窗口** | 1-24 小时 | 通常 24 小时 |
| **概率匹配窗口** | ≤24 小时 | 准确率随时间快速衰减 |

### 3.2 主要平台默认窗口

| 平台 | 点击窗口 | 展示窗口 |
|------|---------|---------|
| Meta (Facebook) | 7 天 | 1 天 |
| Google Ads | 30 天 | - |
| Twitter/X | 14 天（可配置 1-90 天） | 1 天 |

### 3.3 其他窗口类型

- **再营销窗口 (Re-engagement Window)**：流失用户回归归因到再营销活动的时间范围
- **再归因窗口 (Reattribution Window)**：卸载后重新安装被视为新归因的时间范围（AppsFlyer 默认 90 天）

## 4. 归因匹配方式

MMP 按严格的优先级层级进行匹配：

### 4.1 设备 ID 匹配（确定性匹配 — 最高优先级）

```
广告点击 → 传递 IDFA/GAID → MMP 记录
                                    ↓ 精确匹配
应用安装 → SDK 采集 IDFA/GAID → MMP 比对
```

- **iOS**: IDFA（需 ATT 授权，全球授权率约 13-35%）
- **Android**: GAID / AIFA
- **准确率**: 100%

### 4.2 Referrer 匹配（确定性匹配 — 同等优先级）

- 仅限 Google Play，通过 `INSTALL_REFERRER` 参数传递
- 准确率与设备 ID 匹配相当
- 不依赖广告网络传递设备 ID

### 4.3 概率匹配 / 指纹匹配（兜底方案）

- 组合 IP 地址 + User-Agent + 其他设备信号生成"指纹"
- **误差率**: 30-50%（共享 WiFi 或拥挤基站下更高）
- 现代 MMP 使用 ML 概率建模替代原始指纹
- 归因窗口短（通常数小时）
- Apple 已明确反对指纹匹配，可能违反 App Store 政策

### 匹配优先级

```
设备 ID 匹配 (IDFA/GAID)  →  最高优先级
        ↓ 无匹配
Referrer 匹配 (Google Play) →  次优先级
        ↓ 无匹配
概率 / 指纹匹配            →  兜底方案
```

## 5. 核心概念

### 5.1 安装归因 (Install Attribution)

核心流程：
1. MMP SDK 检测新设备（未知设备 ID）
2. 触发归因流程
3. 在归因窗口内扫描匹配的点击/展示记录
4. 归因到获胜的触点

### 5.2 再营销归因 (Re-engagement Attribution)

对**已有用户**的归因。用户点击再营销广告 → 通过深度链接打开 App → 深度链接 URL 携带归因数据。

### 5.3 再归因 (Re-attribution)

用户卸载后重新安装，且此次安装归因到新的广告活动（超出原始安装的再归因窗口）。

### 5.4 深度链接 (Deep Linking)

| 类型 | 说明 |
|------|------|
| **标准深度链接** | 应用已安装时直接跳转到指定页面（URI Scheme / Universal Links / App Links） |
| **延迟深度链接** | 应用未安装时记住目标页面，经过应用商店安装后，首次打开时跳转到指定页面 |

### 5.5 回传 (Postback)

MMP 归因决策后发送给广告网络和广告主的 S2S 回调。

**类型**：
- **安装回传**: 通知广告网络获得安装归因
- **事件回传**: 通知后续事件（购买、注册等）
- **再营销回传**: 通知再营销归因

**回传 URL 结构**：
```
https://postback.example.com/install
  ?clickid={clickid}
  &device_id={aaid}
  &campaign_id={campaign_id}
  &app_id={app_id}
  &revenue={revenue}
  &event_name={event_name}
  &country={country_code}
  &timestamp={timestamp}
```

## 6. 隐私归因框架

### 6.1 ATT 的影响

iOS 14.5+ 的 App Tracking Transparency 要求用户显式授权才能访问 IDFA：
- 全球授权率已降至 13-35%
- IDFA 确定性归因在 iOS 上基本不可用

### 6.2 SKAdNetwork 4.0

Apple 的隐私归因框架，无需用户级标识符：

| 特性 | 说明 |
|------|------|
| **三个回传窗口** | 0-2 天、3-7 天、8-35 天 |
| **转化值** | 细粒度（6-bit，64 个值）/ 粗粒度（低/中/高） |
| **人群匿名性** | 数据粒度随安装量扩展，安装越多数据越详细 |
| **延迟回传** | 最少 24-48 小时 + 随机延迟 |
| **无用户级数据** | 所有归因都是聚合的 |

### 6.3 AdAttributionKit (AAK) — Apple 下一代框架

WWDC 2024 发布，2025 增强：
- SKAdNetwork 的继任者（SKAN 暂未废弃）
- **新增能力**: 再营销归因、多商店支持、可配置归因窗口、国家代码
- 可与 SKAN 4.0 同时运行

### 6.4 趋势展望

- SKAN 5.0 预计提供更快的回传（小时级而非天级）
- 内置增量测试、隐私安全的重定向能力
- IDFA 实质性消亡，基于同意的追踪成为标准
- 差分隐私和 ML 预测建模填补归因空白

## 7. 反作弊机制

### 7.1 主要作弊类型

#### 点击泛滥 (Click Spamming / Click Flooding)
- **手法**: 大量发送虚假点击，当真实用户自然安装时窃取归因
- **也称**: 有机流量窃取 (Organic Poaching)
- **检测**: 异常低的点击-安装转化率，单一来源的超高点击量

#### 点击注入 (Click Injection) — 仅限 Android
- **手法**: 恶意 App 监听 Android 安装广播，在安装完成前注入虚假点击
- **利用**: 末次点击模型的时间漏洞
- **检测**: CTIT（点击到安装时间）为 0-2 秒属于异常。iOS 无此问题

#### SDK 欺骗 (SDK Spoofing)
- **手法**: 逆向 MMP SDK 通信协议，直接向 MMP 服务器发送虚假安装/事件信号
- **影响**: 单个活动中可占比高达 80%
- **检测**: SDK 签名、加密验证、闭源 SDK

#### 设备农场 (Device Farms)
- **手法**: 大量真实设备批量安装，使用设备 ID 重置、VPN 和 GPS 欺骗
- **检测**: IP 分析、设备密度分析、行为分析（低留存、无真实互动）

### 7.2 防御策略

```
第 1 层: SDK 级加密签名
第 2 层: 实时归因前拦截
第 3 层: 归因后欺诈分析
第 4 层: CTIT 分布分析
第 5 层: IP / 地理位置验证
第 6 层: 行为异常检测（留存、互动模式）
第 7 层: 独立第三方反欺诈检测
```

### 8.1 端到端归因数据流

```

       │                        │                          │
  1. 投放广告 ──────────────────>│                          │
       │  (tracking link 携带   │                          │
       │   设备 ID、活动信息)    │                          │
       │                        │                          │
       │              2. 记录点击/展示数据                   │
       │                 (设备 ID、IP、UA、                  │
       │                  时间戳、活动、素材)                 │
       │                        │                          │
       │                        │     3. 用户安装并打开 App ─┤
       │                        │        SDK 发送首次打开    │
       │                        │        (设备 ID、IP、UA)   │
       │                        │                          │
       │              4. 匹配引擎查询点击/展示库              │
       │                 在归因窗口内查找匹配触点              │
       │                        │                          │
       │              5. 做出归因决策                        │
       │                 (设备 ID > Referrer > 概率)         │
       │                        │                          │
  6. 接收安装回传 <──────────────│──────────────────────────>│
       │                        │                          │
       │              7. 存储归因数据，生成报表                │
       │                        │                          │
       │                        │     8. 应用内事件上报 ─────┤
       │                        │        (购买、注册等)      │
       │                        │                          │
  9. 接收事件回传 <──────────────│                          │
       │                        │                          │
```

## 9. 集成方式

### 9.1 SDK 集成

主要集成方式，MMP SDK 嵌入到移动应用中。

**SDK 职责**：
- 首次打开检测
- 设备 ID 采集
- 会话追踪
- 应用内事件上报
- 深度链接处理
- SKAN 转化值管理

**支持平台**: iOS、Android、React Native、Flutter、Unity

### 9.2 Server-to-Server (S2S) 集成

通过 REST API 从后端上报事件，无需 SDK 依赖。

**核心接口**：
| API | 说明 |
|-----|------|
| `SESSION API` | 通知 MMP 应用打开（归因必需） |
| `EVENT API` | 上报应用内事件和收入 |

**适用场景**: 订阅事件、服务端验证的购买、后端触发的事件

**注意事项**：
- 事件需在安装处理完成后发送（安装处理约需 20-30 秒）
- 过早到达的事件可能被标记为自然流量

### 9.3 混合集成（推荐）

- **SDK**: 负责会话管理、归因、深度链接
- **S2S EVENT API**: 负责服务端验证的事件（购买、订阅）
- 大多数 MMP 推荐此方式，兼顾准确性和灵活性

### 9.4 广告网络集成

- **Tracking Links**: 携带宏参数的 URL 模板，广告网络在点击时填充数据
- **Postback Templates**: MMP 在转化时填充并发送给广告网络的 URL 模板
- **SRN 集成**: Meta、Google、TikTok 等自报告网络自行归因，MMP 负责跨网络去重

### 9.5 数据导出

- Raw Data API / Push API：实时或批量发送归因数据
- 常见目标：BigQuery、Snowflake、Redshift、BI 工具、CRM 系统
